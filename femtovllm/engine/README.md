# Engine Architecture & System Design / 引擎架构与系统设计

This document defines the core architecture, component boundaries, and API contracts for the `femtovllm` engine.
本文档定义了 `femtovllm` 引擎的核心架构、组件权责以及调度机制的 API 契约。

---

## 1. Architecture Evolution / 架构演进

Inspired by industry-leading frameworks like vLLM and SGLang, this engine implements modern LLM serving concepts:
本引擎深度融合了现代大模型推理框架的核心理念：

- **Unified Token Scheduling:** Eliminates the strict Prefill/Decode boundary. Everything is treated as "tokens catching up", enabling native Chunked Prefill.
  **统一 Token 调度:** 消除严格的 Prefill/Decode 边界，统一视为“Token 追赶”，原生支持 Chunked Prefill。
- **Lazy Tensorization:** `InputBuilder` only creates lightweight `list[int]`. Tensors are dynamically flattened and built in `ModelRunner` right before execution.
  **延迟张量化:** 输入阶段仅构建轻量级的 `list[int]`，直到进入 `ModelRunner` 前向传播时，才动态展平并构建变长张量（Varlen Tensor）。
- **Prefix-Tree Native Caching (v3):** Natively integrates a block-aligned Prefix Tree into the Scheduler for Zero-Copy prompt caching and dynamic block deduplication.
  **原生前缀树缓存 (v3):** 在调度器中原生集成块对齐的前缀树，实现零拷贝的提示词缓存与动态物理块去重。

## 2. Component Boundaries / 组件权责与状态归属

To prevent state corruption and memory leaks, `Sequence` mutations are strictly isolated across components. `CoreEngine` acts as the central orchestrator.
为防止状态机混乱和内存泄漏，各组件对 `Sequence` 的操作权限被严格物理隔离。`CoreEngine` 充当全局的中央编排器。

| Target / 目标属性 | Component / 组件 | Access / 读写权限 | Description / 状态流转说明 |
| :--- | :--- | :--- | :--- |
| `token_ids` | **CoreEngine** | **Read / Write** | Appends new tokens generated by `ModelRunner` / 接收模型输出并追加新 Token |
| `num_computed_tokens` | **Scheduler** / **CoreEngine** | **Read / Write** | `Scheduler` fast-forwards it during Prompt Caching; `CoreEngine` advances it after forward passes / 调度器在命中缓存时快进该值；引擎在前向传播后推进该值 |
| `status` & `stop_reason` | **Scheduler** / **CoreEngine** | **Read / Write** | `Scheduler` manages lifecycle (`WAITING` <=> `RUNNING`), `CoreEngine` triggers `FINISHED` / 调度器管理队列流转，引擎触发完成状态 |
| `Sequence` | **KVCacheManager** | **Read Only** | Strictly read-only, uses `seq_id` and `num_computed_tokens` for math / 绝对只读，仅读取 ID 和计算进度用于物理块分配 |

## 3. Scheduling as a Multi-Dimensional Knapsack / 多维背包问题与两阶段调度

A scheduling step is a **Multi-Dimensional Knapsack Problem** constrained by Compute Budget (`StepBudget`) and Memory (`KVCacheManager`). The core logic is a two-phase greedy allocation:
调度本质上是一个**多维背包问题**，受限于计算预算（`StepBudget`）和显存（`KVCacheManager`）。其核心逻辑是一个两阶段的贪心分配：

### 📝 Core Scheduling Logic (Pseudo-code) / 核心调度逻辑伪代码

```python
# Phase 1: Schedule Running (Prioritize existing sequences / 优先保障运行中序列)
for seq in running_queue:
    # Fast-forward cached prefixes (v3) / 快进已缓存的前缀
    scheduler.fast_forward_prefix(seq)
    
    # 1. Truncate by computation limit / 受限于 Step 计算预算
    num_tokens = min(seq.uncomputed, budget.remaining)
    
    # 2. Resolve storage limit (KV Cache) / 解决物理块存储限制
    if not kv_cache_manager.can_allocate(seq, num_tokens):
        handle_preemption(seq) # Trigger Preemption / 触发抢占 (OOM 处理)
        continue
            
    allocate_resources(seq, num_tokens)
    scheduled.append((seq, num_tokens))

# Phase 2: Schedule Waiting (Admit new sequences / 在资源充裕时接入新序列)
for seq in waiting_queue:
    # Truncate by BOTH computation and storage / 同时受限于计算量与物理块限制
    num_tokens = min(seq.uncomputed, budget.remaining, kv_cache_allocable)
    
    if num_tokens > 0:
        promote_to_running(seq)
        allocate_resources(seq, num_tokens)
        scheduled.append((seq, num_tokens))
    else:
        break # Stop immediately if any resource is exhausted / 资源耗尽则停止接入
```

## 4. Core API Contracts & Atomic Operations / 核心 API 契约与原子操作

The system relies on strict API contracts between `CoreEngine` and `Scheduler` to guarantee strong consistency between Queue Status and Physical Resources.
系统依赖 `CoreEngine` 与 `Scheduler` 之间严格的 API 契约，以保证**队列状态**与**物理资源**的强一致性。

- **`Scheduler.step()`**
  [Core Logic] Evaluates budgets and block tables. Returns `scheduled` tokens for the model to run, and `aborted` sequences for failed requests.
  [核心逻辑] 评估预算与物理块。返回 `scheduled`（供模型执行的序列及 Token 数）和 `aborted`（因异常被终止的序列）。

- **`Scheduler.commit_blocks(seq)` (v3)**
  [State Bridge] Bridges computation and storage. Must be called *immediately after* a successful forward pass to register newly computed blocks into the Prefix Tree and free redundant physical blocks.
  [状态桥接] 桥接计算与存储。必须在成功的前向传播后**立即调用**，将新计算的物理块注册入前缀树，并释放冗余的物理块。

- **`Scheduler.free_and_finish(seq, stop_reason)`**
  [Atomic] `*` => `FINISHED`. Instantly frees all KV cache blocks, removes the sequence from all queues, and records the `stop_reason`.
  [原子操作] `*` => `FINISHED`。立即释放所有物理块，将序列从队列中剔除，并记录终止原因。

- **`CoreEngine.step()`**
  [Orchestrator] Drives the entire pipeline: `Scheduler.step()` -> `ModelRunner.step()` -> Advance `num_computed_tokens` -> `Scheduler.commit_blocks()` -> Append new `token_id` -> Yield `StepDelta`.
  [全局编排] 驱动整个流水线：获取调度结果 -> 触发 GPU 计算 -> 推进计算进度 -> 提交物理块去重 -> 追加新 Token -> 返回增量结果。

## 5. State Machine Constraints / 状态机流转约束

- **The (L-1) State Healing Taboo / (L-1) 状态自愈禁忌 (v3)**
  During prompt caching, the scheduler forces an `(L-1)` misalignment to guarantee at least one forward pass. It is **STRICTLY FORBIDDEN** to call `commit_blocks()` before this forward pass completes, otherwise the engine will crash due to a `num_blocks_computed < num_blocks_saved` state violation.
  在提示词缓存期间，调度器会强制制造 `(L-1)` 的状态错位以保证至少执行一次前向传播。**严禁**在该前向传播完成前调用 `commit_blocks()`，否则引擎将因状态机违例而崩溃。

- **Immediate Reclaim Mechanism / 即时回收机制**
  When `CoreEngine` detects a stop condition (e.g., hitting `EOS`), it immediately invokes `free_and_finish`. The physical blocks are returned to the allocator in the exact same step.
  当检测到停止条件（如生成了 EOS）时，会立即调用 `free_and_finish`。物理块在当前 Step 就会全额归还给分配器。

- **Lazy Preemption / 延迟抢占**
  When a non-head sequence hits a memory bottleneck, the Scheduler stalls it instead of preempting immediately. Destructive preemption is only triggered when higher-priority requests actually need the physical blocks.
  当非队首序列遇到显存瓶颈时，Scheduler 会优先将其挂起（Stall），直到高优先级请求确实需要物理块时，才会触发破坏性的抢占。
